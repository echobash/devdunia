Why Kafka Exists: The Message Queue Problem

Traditional message queues — RabbitMQ, ActiveMQ, SQS — were built around one core idea: deliver a message, then delete it. A producer sends a job, the queue holds it, a consumer picks it up, the message disappears. Simple. But this model breaks down at LinkedIn scale, where the need was to move 1+ trillion messages per day across dozens of systems simultaneously.

THE LINKEDIN ORIGIN STORY

Jay Kreps, Neha Narkhede, and Jun Rao built Kafka at LinkedIn in 2010. They needed a unified pipeline to handle activity tracking, metrics, log aggregation, and stream processing — all at once. By 2014, LinkedIn was processing over 1 trillion messages per day. Kafka was open-sourced via Apache in 2011 and has since become the de-facto backbone for event-driven architectures at companies like Uber, Netflix, Twitter, and Cloudflare.

Topics and Partitions: The Core Data Model

A topic is a named stream of events — think of it like a database table or a log file category. A topic called user-clicks holds all click events; order-placed holds all orders. But a topic alone isn't enough for scale — Kafka splits each topic into partitions.

TOPIC
  • A logical category name — user-clicks, payments, sensor-data
  • Producers write to a topic by name
  • Consumers subscribe to a topic by name
  • Equivalent to a "database table" but append-only and time-ordered
  • Retention is configured per-topic (time or size based)
PARTITION
  • An ordered, immutable sequence of messages within a topic
  • Each message gets a monotonically increasing offset (0, 1, 2, ...)
  • Stored as a physical log file on disk on a broker
  • Parallelism unit: each partition can be consumed independently
  • Ordering guaranteed within a partition, NOT across partitions

Partition Key and Hash Routing

WITH PARTITION KEY

If a producer sends a message with a key, Kafka computes:

partition = hash(key) % num_partitions

Same key always routes to the same partition. This preserves ordering per key (e.g., all events for user-123 stay in order in partition 0).

WITHOUT KEY (ROUND-ROBIN)

If no key is provided, Kafka distributes messages across partitions in round-robin fashion (or sticky-batch in newer versions for better throughput). Messages are spread evenly but ordering is NOT guaranteed across partitions.

ProducerRecord("topic", null, value)
// key=null → round-robin
ORDERING GUARANTEE: WITHIN A PARTITION ONLY

This is one of the most misunderstood aspects of Kafka. If you need all events for a specific entity (user, order, device) to be processed in order, always send with that entity's ID as the key. Kafka guarantees messages with the same key land in the same partition — and within a partition, offset order = arrival order. Across partitions, there is no global ordering.

Producers: How Messages Get Written

A Kafka producer is a client library that serializes records, determines which partition to write to, batches them up for efficiency, optionally compresses them, and then sends the batch to the partition's leader broker. The producer is designed to be high-throughput and configurable for the reliability level you need.

ACK Levels: Trading Durability for Speed

acks=0 — Fire & Forget

Producer sends and never waits for a response. Maximum throughput, zero durability. If the broker crashes before writing, the message is silently lost. Use only for high-volume metrics where occasional loss is acceptable.

Risk: data loss on broker failure
acks=1 — Leader ACK

Producer waits for the partition leader to write the message to its local log. Replicas may not have it yet. If the leader crashes before replication, message is lost. Good balance of throughput and safety for non-critical data.

Risk: loss if leader fails mid-replication
acks=all (-1) — All Replicas

Producer waits until all ISR replicas (in-sync replicas) have written the message. Strongest durability guarantee. Requires min.insync.replicas to be set (typically 2). Lowest throughput but no data loss even if broker crashes.

Strongest: no loss even on broker failure
BATCHING: THE THROUGHPUT MULTIPLIER

The producer doesn't send each message immediately — it accumulates messages in a per-partition buffer. Two settings control this: batch.size (default 16 KB — flush when buffer fills) and linger.ms (default 0 ms — wait this long for more messages before flushing). Setting linger.ms=5 with compression can increase throughput by 10x at the cost of 5ms additional latency — a common production trade-off.

Consumers and Consumer Groups

A Kafka consumer reads messages from partitions by tracking an offset — a simple integer pointing to the next message to read. It advances the offset as it processes messages, at its own pace. The broker doesn't push; the consumer pulls. This design means a slow consumer never slows down a fast producer.

REBALANCING

When a consumer joins or leaves a group (including crashes), Kafka triggers a rebalance. The group coordinator (a broker) pauses all consumers, reassigns partitions, and resumes. During the rebalance window (can be seconds), no messages are consumed. Cooperative rebalancing (Kafka 2.4+) dramatically reduces this pause by only reassigning partitions that change owners.

PARALLELISM CEILING

The maximum useful parallelism in a consumer group equals the number of partitions. If you have 3 partitions and add a 4th consumer to the group, that consumer sits idle — there are no partitions to assign it. This is why partition count is a key design decision: set it higher than your current consumer count to allow future scale-out without re-partitioning.

Brokers, Leaders, and Replication

A broker is a single Kafka server — a JVM process that stores partition data on disk and serves read/write requests. A Kafka cluster is typically 3–10+ brokers. Partitions are spread across brokers, and each partition has exactly one leader and zero or more follower replicas. All reads and writes go through the leader.

ISR: IN-SYNC REPLICAS

The ISR is the set of replicas that are caught up with the leader (within replica.lag.time.max.ms, default 10 seconds). If a follower falls too far behind, it's removed from the ISR. A write with acks=all only succeeds when all ISR members have acknowledged. If the ISR shrinks to just the leader, min.insync.replicas prevents writes from succeeding if durability is required.

LEADER ELECTION ON FAILURE

When a broker crashes, the partitions it led need new leaders. In KRaft mode (Kafka 2.8+), the cluster controller handles leader election directly. Only replicas in the ISR are eligible to become leader by default (unclean.leader.election.enable=false). Unclean election risks data loss — a lagging replica becomes leader and the messages it missed are gone forever.

Log Segments and Retention

Each Kafka partition is physically stored as a series of log segment files on disk. Kafka doesn't write one giant endless file — it rolls over to a new segment when the current one hits a size limit or age threshold. Only the newest segment (the active segment) accepts new writes; all others are immutable and subject to retention or compaction.

Each .log file is paired with a .index (offset index) and .timeindex (timestamp index) for O(log N) random access.
TIME-BASED RETENTION

Default: log.retention.hours=168 (7 days). Kafka's log cleaner thread scans segments. When the oldest message in a closed segment is older than the retention period, the entire segment file is deleted atomically. Configurable per-topic with retention.ms.

LOG COMPACTION

For changelog topics (e.g., user profile updates), use cleanup.policy=compact. Instead of deleting old segments by time, Kafka keeps only the latest value per key. The result is a compact "current state" of every key — perfect for Kafka Streams state stores or CDC event streams. Null-value messages are "tombstones" that delete a key.

07

Consumer Offset Management and Delivery Semantics

An offset is just a number — the position of the next message to read from a partition. Consumer groups store their offset progress in an internal Kafka topic called __consumer_offsets. When a consumer restarts, it reads its last committed offset and resumes from there. How and when you commit offsets determines your delivery guarantee.

AUTO COMMIT (enable.auto.commit=true)

Every auto.commit.interval.ms (default 5000 ms), the consumer automatically commits the highest fetched offset. Problem: if the consumer crashes between fetching a batch and the 5-second auto-commit, those messages will be re-delivered when it restarts. This is at-least-once delivery (you may process the same message twice after a crash).

MANUAL COMMIT

Disable auto-commit and call commitSync() or commitAsync() only after you've successfully processed the batch. commitSync blocks until the broker ACKs. commitAsync doesn't retry on failure (dangerous). Best practice: use commitAsync in the happy path and commitSync in the shutdown/error handler.

Delivery Semantics: The Big Three

At-Most-Once

Commit offset before processing. If crash occurs during processing, the message is NOT re-delivered — it's skipped forever.

commitOffset(msg.offset)
process(msg) ← crash here?
// message gone forever
Use: metrics you can afford to lose
At-Least-Once

Commit offset after processing. If crash occurs before commit, the message is re-delivered — your consumer must handle duplicates.

process(msg) ← crash here?
commitOffset(msg.offset)
// re-delivered on restart
Use: most production systems (idempotent sink)
Exactly-Once (EOS)

Achieved via idempotent producer + transactional API. Each producer batch has a sequence number; duplicates are deduplicated by the broker. Read-process-write in a single atomic transaction.

producer.initTransactions()
producer.beginTransaction()
process + produce output
sendOffsetsToTransaction()
producer.commitTransaction()
Use: payments, financial records, billing
OFFSET HISTORY: FROM ZOOKEEPER TO KAFKACRAFT

Before Kafka 0.9 (2015), consumer offsets were stored in ZooKeeper — a distributed coordination service that became a bottleneck and operational burden. Kafka 0.9 moved offsets to the internal __consumer_offsets topic (50 partitions by default), making offset commits just Kafka writes. Kafka 2.8 (2021) introduced KRaft mode, replacing ZooKeeper entirely with Kafka's own Raft-based metadata management — eliminating the last external dependency.
