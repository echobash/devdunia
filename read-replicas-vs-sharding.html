The Database Scaling Problem

A single database node handles your first million rows beautifully. Primary key lookups are fast. Writes complete in microseconds. Life is good. Then your product takes off — and suddenly the database is the bottleneck for everything. But what kind of bottleneck? That question determines the entire solution.

Bottleneck Type A: Too Many Reads
SYMPTOMS
SELECT queries piling up CPU at 95% p99 read latency > 500ms
  • Reporting dashboards running heavy analytics
  • API endpoints reading same rows repeatedly
  • Search queries doing full table scans
  • 10x more reads than writes in your workload
Solution: Read Replicas

THE KEY INSIGHT

Read replicas add more read capacity — but every replica still holds ALL the data and the primary still handles ALL writes. Sharding splits the data across multiple primaries — each primary owns a slice. These are complementary, not competing: production systems often combine both (shards with replicas on each shard).

Read Replicas: How They Work

A read replica is an exact copy of the primary database that receives a live stream of every change the primary makes. The primary handles all writes. Replicas apply those changes asynchronously and serve read queries. Your application routes reads to replicas and writes to primary — often via a load balancer or connection pooler like PgBouncer.

Replication Methods

WAL Streaming (PostgreSQL)

PostgreSQL writes every change to its Write-Ahead Log before committing. Replicas connect directly to the primary and stream WAL records in real time. Physical replication: byte-for-byte identical. Logical replication: row-level change stream, supports cross-version.

Binary Log (MySQL)

MySQL's binlog records every data change as an event. Replicas connect and apply binlog events. Three formats: STATEMENT (replays the SQL), ROW (replays the exact row changes — safer), MIXED (auto-picks). GTID-based replication tracks position globally across failovers.

Change Data Capture

Tools like Debezium read the WAL/binlog and publish changes to Kafka. Downstream consumers (Elasticsearch, Redis, data warehouse) subscribe and build their own projections. CDC enables read replicas outside the DB itself — your search index is a "replica".

Replication Lag: The Hidden Problem

Asynchronous replication means the replica is always slightly behind the primary. Under normal conditions this lag is microseconds to milliseconds — imperceptible. Under heavy write load, a slow network, or a replica catching up after restart, lag can grow to seconds. This creates a subtle but dangerous bug: read-your-own-writes inconsistency.

The user updates their profile name and immediately refreshes the page — but the read hits a replica still applying the WAL record. They see the old name. This is the read-your-own-writes problem.

Solutions to Read-Your-Own-Writes

ROUTE POST-WRITE READS TO PRIMARY

After any write, set a flag in the session (or a cookie) and route the next N reads for that user to the primary. Cheap, effective, but adds load to primary for that window.

WAIT FOR REPLICA SYNC POINT

PostgreSQL: pg_current_wal_lsn() returns the write LSN. Pass it to the replica and call pg_wal_replay_wait(lsn) — the read blocks until the replica catches up to that exact log position.

STICKY REPLICA SESSIONS

Hash the user ID to always route to the same replica. That replica's lag is now consistent for that user — they always see a consistent (if slightly old) view. Combined with short TTL helps.

MONITORING LAG

PostgreSQL: SELECT write_lag, flush_lag, replay_lag FROM pg_stat_replication; — shows lag per replica in wall-clock time. MySQL: SHOW SLAVE STATUS\G shows Seconds_Behind_Master. Alert if lag exceeds your SLA. Normal: sub-millisecond. Concerning: > 1 second. Critical: > 10 seconds.

04

Read Replica Use Cases and Limits

Read replicas are the right tool for a specific category of problem: too many read queries hitting one node. Before reaching for sharding (complex, irreversible), exhaust read replicas combined with a caching layer — this solves most production read-scaling problems.

GREAT USES FOR READ REPLICAS
  • Analytics and reporting queries — slow GROUP BY / aggregates that would block writes on primary
  • Dashboard reads — polling every 10 seconds for metrics
  • Search workloads — full-text search queries routed to replica
  • Cross-region latency — serve US-West users from a US-West replica, EU from EU replica
  • Read-heavy APIs — GET endpoints that rarely mutate data
  • Backups without locking primary — take pg_dump from replica
WHAT REPLICAS CANNOT FIX
  • Write throughput — the primary is still the only write node
  • Data volume — every replica holds ALL the data (same storage cost per replica)
  • Write contention — row-level locks, serialization conflicts still happen on primary
SCALING LIMIT: THE PRIMARY WALL

You can add 10 or 20 read replicas. Each replica can serve read traffic independently. But every single write still goes through the one primary. When writes hit the primary's ceiling (typically ~5,000–20,000 TPS for transactional workloads), no amount of read replicas helps. That's the wall where sharding begins.

Replicas: 1 primary → N replicas
All N replicas receive EVERY write
Write capacity = capacity of 1 primary
Read capacity = N × replica read throughput
TYPICAL PRODUCTION STACK
Application Servers route writes vs reads
Redis Cache cache hot reads, TTL 60s

Redis absorbs 80-95% of reads. Replicas handle the rest. Primary sees only writes.

Sharding: How It Works

Sharding splits your data across multiple independent database nodes called shards. Each shard is a full primary that owns a subset of the data — it handles reads and writes for its slice. Add a shard and you add proportionally more write capacity and storage. This is horizontal scaling for writes.

HASH-BASED SHARDING

Apply a hash function to the shard key, take modulo N. Result: even distribution regardless of key values. Best for write throughput. Downside: range queries across many user IDs must fan out to all shards.

shard = crc32(user_id) % num_shards
RANGE-BASED SHARDING

Shard 0 owns user_id 0–249K, shard 1 owns 250K–499K, etc. Range queries are efficient (all data on one shard). Danger: new users go to the latest shard → hot shard problem. Needs careful key design or time-based rotation.

shard = floor(user_id / 250000)
THE CROSS-SHARD JOIN PROBLEM

SQL JOINs between tables on different shards are impossible at the database layer. If users is sharded by user_id and orders is sharded by order_id, fetching a user's orders requires querying both shards and joining in application code. This is the most painful operational consequence of sharding — schema design must minimize cross-shard queries.

Shard Key Selection: The Most Critical Decision

The shard key is the column used to decide which shard owns a row. Choose wrong and you get a hot shard — one node gets all the traffic while the others sit idle. This is worse than no sharding at all. Choosing the shard key is often the hardest architectural decision in distributed databases.

Bad Shard Key: signup_date
Range shard: shard 0 = 2020, shard 1 = 2021,
shard 2 = 2022, shard 3 = 2023–now

Key Selection Rules

Property Requirement Why It Matters
High Cardinality Millions of unique values minimum Low cardinality (e.g. country_code) gives you at most N distinct shards where N = distinct values. 3 countries = 3 shards maximum.
Even Distribution Roughly equal row counts per shard Uneven distribution = hot shards. US users might be 10× more than other countries — shard by country puts 10× load on US shard.
Query Locality Most queries touch rows in one shard If a user's feed, posts, and messages all live on their shard, most queries are shard-local. Cross-shard scatter-gather kills latency.
Immutable Key value never changes after insert If a user can change their user_id (unlikely) or you shard by email (changeable), updating a row means deleting from one shard and inserting into another — a nightmare.
REBALANCING: THE NIGHTMARE

Using hash(key) % N — if you go from 4 to 5 shards (N changes), 75% of all keys remap to a different shard. That means moving 75% of your data while the system is live under load. This is why consistent hashing was invented — it reduces remapping to ~1/N of keys when adding one shard. See Section 07.

07

Consistent Hashing

Standard modulo hashing breaks when you change the number of shards. Consistent hashing solves this by placing both keys and nodes on a conceptual hash ring (a circle of values from 0 to 2³²). A key is assigned to the first node clockwise from its position on the ring. Adding a node only moves the keys between that node's predecessor and the new node — roughly 1/N of total keys.

Problem: Naive hash(key) % N
4 shards: key "alice" → shard 2
Add 1 shard (N=5): key "alice" → shard 4 ← moved!
Add 1 shard (N=6): key "alice" → shard 0 ← moved again!
DATA MOVEMENT FORMULA
Keys that move = (N-1)/N × total keys
N: 4→5: moves 80% of all keys
N: 4→6: moves 83% of all keys
Live migration of 80%+ data = production disaster
Solution: Consistent Hashing Ring
4 nodes on ring: key "alice" → Node 2
Add Node 5 between Node 2 and Node 3
Only keys between Node 2 and Node 5 move
~20% of keys move (just 1/N of total)
DATA MOVEMENT FORMULA
Keys that move = 1/N × total keys
N: 4→5: moves ~20% of keys
N: 4→6: moves ~16% of keys
Only the new node's arc needs migration

Virtual Nodes

THE PROBLEM WITH BASIC CONSISTENT HASHING

With 4 nodes placed at fixed hash positions, each node's arc might be very different sizes by chance — one node gets 40% of keys, another gets 5%. Virtual nodes solve this: each physical node is assigned multiple positions on the ring (typically 100–300 virtual nodes per physical node). The arcs average out to roughly 1/N each, giving even load distribution even with a small number of physical nodes.

Physical nodes: 4
Virtual nodes per physical: 150
Points on ring: 600
Distribution variance: <2%
Used by: Apache Cassandra,
Amazon DynamoDB, Riak,
