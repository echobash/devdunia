Before any EDA conversation, you need to speak the language precisely. There are three fundamental message types, and confusing them is the #1 source of poorly-designed event-driven systems. They are not interchangeable.

THE CQS FOUNDATION

Command-Query Separation (CQS) was coined by Bertrand Meyer: every method should either be a command (changes state, returns nothing) or a query (reads state, no changes). CQRS (Command Query Responsibility Segregation) scales this to the architectural level — separate models for writes and reads. Events are what connect those two worlds asynchronously.

PAST TENSE = EVENT IMPERATIVE = COMMAND QUESTION = QUERY

The Event-Driven Flow: Loose Coupling at Scale

The core idea: instead of Service A calling Service B directly (tight coupling), A emits an event into a broker. B, C, D subscribe and react independently. A doesn't know or care who's listening. This single shift solves dozens of distributed systems problems at once.

TEMPORAL DECOUPLING

The producer emits an event and does not wait for consumers to react. The event is persisted in the broker. Consumers process it at their own pace — even if they're down when the event is published, they'll catch up when they come back online. This breaks the "both services must be up simultaneously" constraint of synchronous calls.

LOCATION DECOUPLING

The producer has no idea who consumes its events. You can add a new consumer (analytics, audit log, notification service) without touching the producer at all. Removing a consumer requires zero changes to anything else. This is the key to independent deployability in microservices.

Event Sourcing: The Log Is the Truth

Traditional systems store current state: you update a row, the old value is gone. Event Sourcing inverts this: you store the sequence of events that led to the current state. Current state is derived by replaying events from the beginning. This is how your bank account works — the balance isn't stored directly; it's the sum of all deposits and withdrawals.

BENEFITS
  • Full audit log: every change is recorded, who did what and when — free
  • Time-travel queries: rebuild state at any point in history
  • Rebuild projections: add a new read model by replaying all events
  • Debug by replay: reproduce any bug by replaying events up to that point
  • Event-driven naturally: events are the source of truth, not derived
DRAWBACKS
  • Schema evolution: old events must stay readable as schemas change
  • Replay performance: millions of events = slow cold replay without snapshots
  • Eventual consistency: projections may lag behind the event log
  • Steep learning curve: CRUD instincts don't apply
  • Snapshots needed: periodic state snapshots to avoid full replay
SNAPSHOTS: THE PERFORMANCE FIX

After a million events, replaying from scratch takes too long. Solution: periodically save a snapshot of the current state alongside the event stream. On next load, start from the latest snapshot and replay only events after that snapshot. You get full history without O(n) startup cost. Most event sourcing libraries (Axon, EventStoreDB) handle snapshots automatically.

Choreography vs Orchestration

Two fundamental patterns for coordinating multiple services in an event-driven flow. Both work — but they make very different trade-offs in coupling, debuggability, and who owns the business logic.

Choreography

Each service knows what to do when it sees a specific event. No central coordinator — services "dance" together by reacting to each other's events.

OrderSvc emits OrderPlaced
InventorySvc reacts → emits StockReserved
PaymentSvc reacts → emits PaymentCharged
ShippingSvc reacts → emits Shipped
+ No central point of failure
+ Services fully independent
- Hard to trace: "what is the overall flow?"
- Logic scattered across services
Orchestration

A central saga orchestrator sends commands and waits for responses. It owns the business flow and tracks state explicitly.

OrderSaga Orchestrator
1. ReserveInventory
2. ChargePayment
3. ShipOrder
+ Easy to trace: one place, one flow
+ Business logic is centralized
- Orchestrator is a central point of failure
- Services coupled to orchestrator protocol
WHEN TO CHOOSE WHICH
USE CHOREOGRAPHY WHEN
  • Services are truly independent domains
  • The flow is simple (2-3 steps)
  • Maximum decoupling is the priority
  • Teams own services end-to-end
USE ORCHESTRATION WHEN
  • Flow has complex conditional logic
  • You need clear visibility into saga state
  • Compensating transactions are complex
  • Central ops team manages the flow

The Saga Pattern: Distributed Transactions Without 2PC

In microservices, you cannot do a traditional ACID transaction spanning multiple services — they have separate databases. The Saga pattern solves this: a saga is a sequence of local transactions, each publishing an event. If any step fails, compensating transactions undo the previous steps. Not a rollback — a forward correction.

BOOK TRIP SAGA — HAPPY PATH vs COMPENSATION
Happy Path
1. Book Hotel
emits HotelBooked

emits FlightBooked

no cars available

Cancel Flight Booking

KEY INSIGHT: COMPENSATING TRANSACTIONS

Compensating transactions are not rollbacks — you can't un-send an email or un-charge a card. They're semantic undos: a charge becomes a refund, a booking becomes a cancellation. Each compensating transaction is itself a local ACID transaction that emits its own event. The saga tracks which steps completed and which compensations are still pending. This is why saga state must be durable — if the saga coordinator crashes mid-compensation, it must resume correctly.

Event Schema and Evolution

Events are contracts between services. Once consumers depend on an event schema, changing it can break them silently in the worst possible way — at runtime, hours after deployment. Event schema management is the operational tax you pay for the decoupling benefits of EDA.

SCHEMA REGISTRY (CONFLUENT / AVRO)

Instead of ad-hoc JSON, use Apache Avro with a Confluent Schema Registry. Producers register their schema; consumers fetch it by ID embedded in each message. The registry enforces compatibility rules: BACKWARD (new schema reads old data), FORWARD (old schema reads new data), or FULL (both directions). Schema violations are caught at publish time, not at consume time.

Message bytes: [magic byte][schema id 4b][avro payload]
Consumer fetches schema by id → deserializes safely
CONSUMER-DRIVEN CONTRACT TESTING

Each consumer publishes a Pact contract describing exactly which fields it needs from each event. The producer's CI pipeline runs all consumer contracts as tests — if a proposed schema change breaks a consumer's contract, the build fails before the change is deployed. The consumer, not the producer, defines the interface. This inverts the ownership model and prevents breaking changes from ever reaching production.

RULE OF THUMB: ADDITIVE ONLY

The golden rule of event evolution: only add optional fields, never remove or rename. When you must remove a field, run old and new schemas simultaneously during a migration period (dual-write), give all consumers time to update, then retire the old schema. Version your event types in the type name (OrderPlaced.v2) or in a version field so consumers can route accordingly. Treat event schemas like database columns — once deployed, they live forever.

SYNC MODE FAILURE CHARACTERISTICS
  • One service down = client gets an error
  • Partial completion is dangerous (order placed but not paid)
  • Upstream services are blocked waiting for a dead downstream
  • Retry storms: every retry re-executes all steps
  • No work is saved if the chain breaks mid-way
EVENT MODE FAILURE CHARACTERISTICS
  • One consumer fails = event stays in broker, retried later
  • Other consumers keep processing — partial success is fine
  • No cascade: producer never knew about consumer failure
  • Dead letter queue catches poison messages after N retries
  • Service restarts catch up by replaying from broker offset
