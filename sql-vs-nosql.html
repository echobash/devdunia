01

The Core Mental Model Difference

Before benchmarks and trade-offs, there is a philosophical difference. SQL assumes your data has structure — tables, rows, relationships, and rules. NoSQL assumes your data is flexible — documents, keys, graphs, or columns that can evolve as your application grows. Neither is better; they solve different problems for different worlds.

SQL — THE STRUCTURED CITY
users
orders
products
payments
users.id ──FK── orders.user_id
orders.id ──FK── order_items.order_id
products.id ──FK── order_items.product_id
  • Every row in a table follows the same fixed schema
  • Relationships enforced by foreign keys
  • JOINs combine data across tables at query time
  • Schema changes require migrations
  • Consistency guaranteed — ACID transactions
VS
NoSQL — THE FLEXIBLE WORKSPACE
{ "_id": "u001",
"name": "Alice",
"orders": [{ ...nested }],
"prefs": { any shape }
}
  • Each document can have a different shape
  • Nested objects — no JOINs needed for reads
  • Add new fields without a migration
  • Scale out horizontally — add more nodes
  • Availability + partition tolerance prioritised
THE GOLDEN RULE

Choose SQL when your data has complex relationships and consistency is non-negotiable. Choose NoSQL when you need massive horizontal scale, flexible/evolving schemas, or simple access patterns at high throughput. Many production systems use both — PostgreSQL for transactions, Redis for caching, Elasticsearch for search.

SQL databases are built on the relational model introduced by Edgar Codd in 1970. Data lives in tables. Tables relate via keys. The query language (SQL) is declarative — you say what you want, not how to get it. The engine figures out the optimal execution plan.

ACID Properties — The Guarantee

A — Atomicity

A transaction is all-or-nothing. Transfer $100 from Alice to Bob: both the debit and credit succeed together, or neither happens. No partial updates.

BEGIN;
UPDATE accounts SET bal=bal-100 WHERE id=1;
UPDATE accounts SET bal=bal+100 WHERE id=2;
COMMIT; -- both or neither
C — Consistency

Every transaction takes the database from one valid state to another valid state. Constraints (NOT NULL, UNIQUE, FK, CHECK) are enforced. Invalid data is rejected.

INSERT INTO orders(user_id) VALUES (999);
ERROR: FK constraint violated
-- database stays consistent
I — Isolation

Concurrent transactions see each other as if they ran sequentially. Levels: READ UNCOMMITTED, READ COMMITTED, REPEATABLE READ, SERIALIZABLE. Higher isolation = less concurrency.

Tx A reads balance: $100
Tx B writes balance: $50
Tx A re-reads: still $100
-- REPEATABLE READ level
D — Durability

Once a transaction commits, it's permanent. Even if the server crashes immediately after, the data survives. Achieved via WAL (Write-Ahead Log) — write to log first, then to data files.

COMMIT; -- fsync'd to WAL
SERVER CRASHES
Restart: data recovered from WAL
SQL EXAMPLES & BEST USE CASES
POPULAR ENGINES
  • PostgreSQL — open source, full-featured, JSON support, best overall
  • MySQL / MariaDB — dominant in web hosting, fast reads
  • SQLite — embedded, zero config, one file
  • SQL Server — Microsoft ecosystem, BI tooling
  • Oracle — enterprise, stored procedures, extreme reliability
BEST USE CASES
  • Banking / Finance — ACID mandatory, complex transactions
  • E-commerce — orders, inventory, payments, users
  • ERP / CRM — many related entities, reporting
  • Healthcare — patient records, regulatory compliance
  • Analytics — complex aggregations, GROUP BY, window functions

ACID vs BASE: Two Consistency Philosophies

ACID and BASE are not just acronyms — they represent opposing philosophies about the trade-off between consistency and availability. The CAP theorem (Consistency, Availability, Partition tolerance — pick 2) explains why these philosophies exist at all.

ACID — THE RIGID CONTRACT
Atomicity
All ops commit or all rollback
Consistency
Constraints always valid
Isolation
Concurrent txns don't interfere
Durability
Committed data survives crashes
Failure scenario: Bank transfer of $500 fails mid-way. ACID rollback ensures no money vanishes. Both accounts return to original state immediately.
Think: bank vault with a strict ledger.
Every transaction signed and sealed.
BASE — THE FLEXIBLE AGREEMENT
Basically Available
System stays online, even partially
Soft State
State may change without new input (replication settling)
Eventually Consistent
All replicas converge eventually
Failure scenario: You like a tweet. Your like count shows 999 on one replica and 1000 on another for a few seconds. Eventually both agree on 1000. The system never goes down.
Think: a shared whiteboard that syncs
across offices with a small delay.
CAP THEOREM — WHY YOU MUST CHOOSE

A distributed system can guarantee at most 2 of 3: Consistency, Availability, Partition Tolerance. Network partitions always happen in distributed systems, so the real choice is CP vs AP.

  • CP (Consistent + Partition-tolerant): HBase, Zookeeper, MongoDB in strict mode — may be unavailable during partition
  • AP (Available + Partition-tolerant): Cassandra, DynamoDB, CouchDB — always responds, may return stale data
  • CA (Consistent + Available): Only possible without partitions — traditional single-node SQL

Scaling: Vertical vs Horizontal

Scaling is where SQL and NoSQL diverge most dramatically in practice. SQL databases were designed in an era of single powerful servers. NoSQL was designed for the distributed, commodity-hardware world of cloud computing.

SQL — VERTICAL SCALING

  • Simple — no application changes needed
  • Read replicas help with read-heavy workloads
  • Hard ceiling: biggest available server on AWS = ~24TB RAM
  • Downtime required for major upgrades
  • SQL sharding (Vitess, Citus) exists but requires significant engineering
Cost per GB of RAM doubles every 2x
at extreme sizes. Diminishing returns.
NoSQL — HORIZONTAL SCALING
  • Add more commodity servers (nodes) to the cluster
  • Data automatically resharded across nodes (Cassandra ring)
  • Linear throughput growth: 2x nodes = ~2x throughput
  • No ceiling — Netflix runs thousands of Cassandra nodes
  • No single point of failure — nodes replicate to each other
  • New nodes added without downtime
10 × $500 commodity servers >>
1 × $50,000 behemoth server.
NEWSQL: THE BEST OF BOTH WORLDS

NewSQL databases (CockroachDB, Google Spanner, TiDB, YugabyteDB) are a third category: they offer full SQL semantics and ACID transactions with automatic horizontal sharding. They achieve this via distributed consensus protocols (Raft, Paxos). Trade-off: higher latency than a single-node SQL DB, more complexity than simple NoSQL, but globally distributed ACID is finally possible.

SQL STRENGTHS
  • Complex multi-table queries: JOIN 10 tables in one SELECT, the planner optimises execution
  • Data integrity: FK constraints, UNIQUE, CHECK — invalid data rejected at write time
  • Ad-hoc queries: business analysts can query any pattern without pre-defined access patterns
  • Transactions across entities: debit one account, credit another — atomically
  • Aggregation power: SUM, AVG, GROUP BY, ROLLUP, window functions, CTEs
  • Ecosystem maturity: 50 years of tooling — ORMs, migration tools, monitoring, backup solutions
REAL-WORLD SQL USE CASES
  • Banking / Fintech: account balances, double-entry bookkeeping, audit trails
  • E-commerce: order management, inventory, cart, payments (Shopify uses MySQL)

  • Airbnb: listings, bookings, payments — ACID requirements everywhere
SQL WEAKNESSES
  • Schema changes require migrations — painful at scale
  • Vertical scaling hits a ceiling
  • Horizontal sharding is manual and complex
  • Object-relational impedance mismatch (ORM complexity)

When NoSQL Wins

NoSQL databases were born out of necessity — Google's Bigtable (2006), Amazon's Dynamo (2007), and Facebook's Cassandra (2008) were all built because no SQL database could handle their scale, access patterns, or uptime requirements.

Flexible Schema

Startup iterating fast. Product schema changes weekly. With NoSQL, just add a new field to new documents. No ALTER TABLE, no migration scripts, no downtime.

// week 1: {"name":"X"}
// week 3: {"name":"X","tags":[]}
// week 6: {"name":"X","ai_score":0.9}
// all coexist, no migration
Massive Scale

Billions of records, millions of writes per second. Cassandra scales linearly — double the nodes, double the throughput. DynamoDB handles any traffic with no capacity planning.

Netflix: Cassandra for watch history
100TB+ data, 99.9999% uptime
Writes from 190 countries simultaneously
Simple Access Patterns

If your query is always "get user by ID" or "get cart by session", a key-value store does this 10-100x faster than SQL. You don't need JOINs when you never need them.

GET cart:session-abc123
< {"items":[...]} -- 0.1ms
vs SQL JOIN: ~10ms
Time-Series / IoT

1000 sensors writing every second = 86 million rows/day. Column-family DBs are optimized for time-range scans and high write throughput. Cassandra handles this natively.

SELECT * FROM sensors
WHERE id='s001'
AND time > now() - 1h
Graph Relationships

Friend-of-a-friend recommendations, fraud ring detection. In SQL, a 6-hop traversal requires 6 self-JOINs and becomes exponentially slow. Neo4j traverses millions of hops per second.

MATCH (u)-[:KNOWS*1..6]->(rec)
WHERE u.id = 'alice'
RETURN rec LIMIT 10
Global Distribution

Multi-region writes with low latency. Cassandra, DynamoDB Global Tables, and Cosmos DB can accept writes in any region and replicate globally. SQL multi-master is complex and rare.

Write: us-east-1 < 1ms
Replicates to: eu-west-1
Replicates to: ap-southeast-1
REAL-WORLD NoSQL EXAMPLES
  • MongoDB: Atlassian product catalogs, Foursquare location data
  • DynamoDB: Amazon shopping cart (never go down during Prime Day), gaming leaderboards
  • Cassandra: Netflix watch history, Instagram timeline, Discord message history
  • Redis: Twitter timeline cache, Slack presence data, GitHub rate limiting
  • Elasticsearch: GitHub code search, Uber driver search
  • Neo4j: LinkedIn "People You May Know", eBay fraud detection graph
  • HBase: Facebook messages (2.5 billion users), Apple iCloud metadata
  • Firestore: Google Workspace real-time collaboration, mobile app backends

Interactive: Schema Design Comparison

Pick a scenario and see how SQL and NoSQL design the same data differently. SQL normalizes into multiple tables linked by foreign keys. NoSQL embeds related data into a single document. Each has real trade-offs.

SCHEMA DESIGN — SQL vs NoSQL Comparison
SCENARIO
USER PROFILE + POSTS
SQL: 3 tables (users, posts, tags), 2 JOINs to get a user's posts with tags.

NoSQL: 1 document per user with embedded posts array. Fetched in one read, no joins.
TRADE-OFFS
SQL: Normalized — no duplication. Update username once. But needs 2 JOINs for every post page load.

NoSQL: Denormalized — username duplicated in every post. Fast reads but updating username requires updating all documents.
THE NORMALIZATION VS DENORMALIZATION DECISION
SQL: NORMALIZE
  • Store each fact once, reference via FK
  • Update one row to change a value everywhere
  • JOIN at query time to reconstruct the data
  • Ideal when data is updated frequently
NoSQL: DENORMALIZE
  • Embed related data inside the parent document
  • One document fetch = complete data, no JOINs
  • Data duplication is intentional and accepted
  • Ideal when data is read far more than written
NORMALIZE FOR WRITES DENORMALIZE FOR READS CHOOSE WISELY
