01

What CAP Theorem Says

Proposed by Eric Brewer in 2000 and proved by Gilbert & Lynch in 2002, the CAP Theorem states that a distributed data store can guarantee at most 2 of 3 properties simultaneously: Consistency, Availability, and Partition Tolerance.

Consistency (C)

Every read receives the most recent write or an error. All nodes in the cluster see the same data at the same time. There is a single, globally consistent view of the data.

Write X=5 on Node A
Read from Node B
Result: X=5 (guaranteed)
Availability (A)

Every request receives a response — not necessarily the most recent data. Every non-failing node always returns a result (no timeouts or errors), even if data may be stale.

Write X=5 on Node A
Read from Node B
Result: X=3 (old value, but responds)
Partition Tolerance (P)

The system continues operating even when network messages are lost or delayed between nodes. The system survives a split where nodes cannot communicate with each other.

(network partition)
System still handles requests

02

Why Partition Tolerance Is Non-Negotiable

A network partition is when two or more nodes in your cluster cannot communicate with each other — even though both nodes are running fine. The network between them is broken, delayed, or dropping packets.

WHEN PARTITIONS HAPPEN
  • Network cable physically severed
  • Switch or router reboots mid-traffic
  • AWS Availability Zone goes down
  • Packet loss causes timeout between nodes
  • Datacenter network congestion drops packets
  • Firewall rule change cuts inter-node traffic
THE FUNDAMENTAL RULE

If you run your system on more than one node, you will experience partitions eventually. It's not a matter of if — it's a matter of when. This is why P is not optional: you have to decide how your system behaves when a partition occurs.

WHY YOU CAN'T AVOID THIS CHOICE

If Node B receives a read request during a partition, it has exactly two options: (1) refuse to answer and return an error — sacrificing availability but maintaining the consistency guarantee; or (2) answer with the data it has locally — maintaining availability but potentially returning stale data. There is no third option. This is the inescapable trade-off at the heart of distributed systems.

03

CP Systems: Consistency over Availability

A CP system prioritizes correctness: during a network partition, it will refuse to return potentially stale data, instead returning an error. The guarantee: every read returns the most recent write — or fails.

CP BEHAVIOR DURING PARTITION
  • Node cannot confirm write with majority? Reject the write.
  • Node cannot confirm read is fresh? Return error.
  • Uses leader election (Raft/Paxos): minority partition stops accepting writes
  • Client sees: connection refused / timeout / "no leader" error

When to Choose CP

  • Financial transactions — "Did my bank transfer go through?" must be definitive
  • Distributed locks — only one node should hold the lock at a time
  • Leader election — exactly one leader, no split-brain
  • Configuration management — all services must see the same config version
  • Inventory management — overselling a product is a business disaster

Real-World CP Systems

ZooKeeper

Distributed coordination service. Uses ZAB (Zookeeper Atomic Broadcast) for consensus. During partition minority, nodes stop accepting writes. Used by Kafka, Hadoop for leader election.

etcd

Kubernetes' backing store for all cluster state. Uses Raft consensus. A write must be acknowledged by a majority before it's confirmed. Minority partition returns errors.

HBase

Distributed column store (Hadoop ecosystem). Uses HDFS replication and ZooKeeper for region server coordination. Consistency is always guaranteed — writes wait for replication.

04

AP Systems: Availability over Consistency

An AP system prioritizes staying online: during a network partition, every node continues to answer requests based on its own local state — even if that data is stale. The system stays up, but you may read old data. Once the partition heals, nodes reconcile their diverged states — a process called eventual consistency.

AP BEHAVIOR DURING PARTITION
  • Every node answers every request — no errors during partition
  • Reads may return stale data (from before the partition)
  • Diverged writes are reconciled after partition heals
  • Conflict resolution: last-write-wins, CRDTs, or manual merge
EVENTUAL CONSISTENCY

The guarantee: if no new writes happen, eventually all replicas will converge to the same value. "Eventually" could be milliseconds or seconds depending on network conditions. The key word is eventually — not immediately.

Real-World AP Systems

Cassandra

Every node can accept reads and writes. No single point of failure. Replication is asynchronous. Consistency level is tunable per-query (more on this in Section 06).

DynamoDB (default)

Default eventually consistent reads. Maximum availability and low latency. You can opt into strongly consistent reads per request — at higher cost and latency.

DNS

Classic AP system. A DNS record update may take minutes to hours to propagate globally. Every resolver answers immediately with cached data. Eventual consistency by design.

WHEN TO CHOOSE AP
  • Social media likes/counts: seeing 10,203 vs 10,204 likes is not critical
  • Shopping cart: briefly showing stale cart contents is acceptable
  • User profile / settings: a few seconds of staleness is fine
  • DNS records: propagation delay is a known and accepted trade-off
  • Analytics dashboards: near-real-time is good enough
05

CA Systems: The Impossible Category

A CA system provides Consistency and Availability but sacrifices Partition Tolerance. The catch: this is only possible if you can guarantee there will never be a network partition — which is only achievable on a single node.

SINGLE-NODE = CA (SORT OF)

A traditional RDBMS (PostgreSQL, MySQL) on a single server provides both C and A: every read sees the latest committed write, and the server always responds. But it's not a "distributed" system — there's only one node, so partitions can't happen. Once you add replicas, you re-enter CAP territory.

Single PostgreSQL / MySQL
Consistent + Available
But: one SPOF, can't scale horizontally
NEWSQL: CP WITH HIGH AVAILABILITY

Systems like Google Spanner and CockroachDB are often marketed as "CA" but are more accurately CP with very high availability. They use consensus protocols (Paxos / Raft) to achieve strong consistency across nodes. During a partition, the minority side stops accepting writes — but in practice, they're engineered to make partitions extremely rare and brief.

Google Spanner

Globally distributed, externally consistent. Uses TrueTime (GPS + atomic clocks) to order transactions. CP in practice — partition minority halts.

CockroachDB

Distributed SQL with serializable isolation. Uses Raft consensus. CP system — provides ACID transactions across nodes with strong consistency.

THE NUANCE: "CA" IS MOSTLY A FICTION IN DISTRIBUTED SYSTEMS

When system designers say a distributed system is "CA," they typically mean one of two things: (1) it runs on a single node (so partitions can't occur by definition), or (2) it's actually CP but engineered to maximize availability — meaning it's so reliable that partitions are extremely rare, but when they do occur, it still chooses consistency. There is no truly distributed CA system — the network eventually fails.

PACELC: The CAP Extension

CAP only describes behavior during a network Partition. But what about when things are running normally? The PACELC model (Daniel Abadi, 2012) extends CAP to also address the trade-off during normal operation:

THE PACELC FORMULA
P Partition? then C or A — E Else (normal) then L or C

Partition — Availability — Consistency — Else — Latency — Consistency

Even without partitions, replicating data across nodes introduces a choice: you can respond immediately (low Latency, possibly stale) or wait for all replicas to confirm (high Consistency, higher latency).

THE LATENCY/CONSISTENCY TRADE-OFF

Even when the network is healthy, synchronous replication to all replicas before returning to the client adds round-trip latency. To return quickly, you must accept that some replicas may not yet have the latest write — a consistency window exists.

EL (Else Latency): acknowledge write as soon as primary stores it. Replicas sync async. Fast but window of inconsistency.
EC (Else Consistency): wait for all replicas to confirm before returning. Strongly consistent but higher latency on every write.
System Partition Else PACELC
DynamoDB PA EL PA/EL
Cassandra PA EL PA/EL
Spanner PC EC PC/EC
etcd PC EC PC/EC
MongoDB (def) PC EL PC/EL
MySQL (repl) PC EL PC/EL
CouchDB PA EL PA/EL
WHY PACELC MATTERS MORE IN PRACTICE

Network partitions are rare events. The latency/consistency trade-off (the EL vs EC dimension) affects every single operation your system handles. For high-throughput systems, this often matters more day-to-day than the partition behavior. A system like Spanner pays higher latency on every write to guarantee consistency; DynamoDB pays lower latency but accepts a window of inconsistency on every write.
