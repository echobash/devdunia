Two Mental Models: Work Orders vs Whiteboards

Before any code, you need the right mental model. A message queue and an event stream look superficially similar — both move data from one process to another — but they solve fundamentally different problems and make opposite trade-offs around who owns the message and for how long.

THE ONE SENTENCE THAT SEPARATES THEM

In a message queue, consumption is destructive — the act of reading removes the message. In an event stream, consumption is non-destructive — reading just advances your pointer, the data remains for everyone else and for replay.

How Message Queues Work (RabbitMQ Deep Dive)

RabbitMQ is the canonical message queue. Its architecture follows AMQP: producers send to an exchange, which routes messages to one or more queues based on routing keys and binding rules. Consumers register on queues and receive pushed messages.

PUSH MODEL + ACKNOWLEDGEMENT

RabbitMQ pushes messages to connected consumers. The broker tracks which messages are in-flight (delivered but not yet ACKed). When the consumer calls basicAck, the broker permanently removes that message. If the consumer dies without ACKing, the broker re-queues the message for another worker — this is exactly why the push model with per-message ACK provides at-least-once delivery.

COMPETING CONSUMERS

Multiple workers subscribing to the same queue get different messages — the broker hands each message to exactly one worker. This is the core load-balancing pattern: add more workers to a queue when you need more throughput. They compete for work. This is perfect for job dispatch but means no worker ever sees what another consumed.

DEAD LETTER QUEUE (DLQ)

When a message is rejected (NACK), exceeds its TTL, or hits the queue length limit, RabbitMQ routes it to a configured Dead Letter Exchange which delivers it to a DLQ. Your ops team monitors the DLQ for failed jobs — they inspect failures, fix bugs, and manually replay messages from the DLQ. Without a DLQ, failed messages just vanish. Always configure one in production.

How Event Streams Work (Kafka Deep Dive)

Apache Kafka is an append-only distributed log. Events are written to topics which are split into partitions. Every message gets an immutable offset — its permanent position in the log. Consumers are stateful: they remember their position and poll for new messages at their own pace. The broker never pushes.

PULL MODEL

Kafka consumers poll — they call consumer.poll(timeout) in a loop. The broker never pushes. This means slow consumers can't be overwhelmed; they just fall behind. The consumer controls its own pace. If a consumer is offline for a week, it picks up where it left off (within retention limits).

LOG RETENTION + REPLAY

Messages are kept on disk for a configured retention period (default 7 days) or up to a size limit. Any consumer can rewind its offset to 0 and re-process the entire history. This is transformative: you can deploy a new microservice today and replay 3 months of historical events to build its initial state.

PARTITIONS: THE KEY TO KAFKA SCALE

Each topic is split into N partitions. Ordering is guaranteed within a partition (not across). Each partition is an independent log. Within a consumer group, each partition is assigned to exactly one consumer — so if you have 6 partitions and 3 consumers, each consumer handles 2 partitions. Scale consumers up to partition count for maximum parallelism.

The Critical Differences: Deep Comparison

These two patterns look similar on the surface — both move data between processes — but they differ fundamentally on almost every dimension that matters in production. The right choice depends entirely on which properties your use case requires.

Dimension Message Queue (RabbitMQ / SQS) Event Stream (Kafka / Kinesis)
Consumption model DESTRUCTIVE Message is removed after one consumer reads it NON-DESTRUCTIVE Message stays; consumer advances its offset pointer
Delivery model PUSH Broker pushes to subscribed consumers (prefetch count controls flow) PULL Consumer polls on its own schedule; broker waits passively
Message ordering FIFO per queue. Add multiple workers and ordering is lost across workers. Strict ordering within a partition. Use partition keys to preserve order for a user/entity.
Replay NO Once consumed and ACKed, the message is gone forever YES Seek to any offset, rewind to beginning, replay entire history
Consumer model Competing consumers: N workers share M messages. One message = one worker. Consumer groups: group A and group B each get a full independent copy of the stream
Message retention Messages live until consumed (or TTL expires). Queue depth grows if consumers are slow. Messages live until retention policy (e.g. 7 days / 100 GB). Consumed or not — doesn't matter.
Backpressure Queue depth. If workers can't keep up, queue grows and may OOM the broker. Consumer just falls behind in offset. Broker is not affected — data sits safely on disk.
Throughput High throughput for individual jobs. AMQP overhead per message. Typical: tens of thousands/sec. Extremely high throughput via sequential disk writes and batching. Millions of events/sec.
Best for Task queues, job dispatch, request/reply, work distribution across N workers Event sourcing, audit logs, analytics pipelines, microservices choreography, replay
Examples RabbitMQ, ActiveMQ, AWS SQS, Azure Service Bus, Google Cloud Tasks Apache Kafka, AWS Kinesis, Azure Event Hubs, Apache Pulsar, Redpanda

Failure Recovery: The Biggest Practical Difference

MQ Consumer Crash
  1. Broker delivers message M to Worker 1
  2. Worker 1 crashes mid-processing — never ACKs
  3. Broker detects connection drop
  4. Message M is re-queued automatically
  5. Worker 2 picks it up and retries
  6. Worker 2 ACKs — message gone
Result: at-least-once delivery. Message is retried automatically. You may get duplicates — make your handler idempotent.
Stream Consumer Crash
  1. Consumer reads offset 5, crashes without committing offset
  2. Consumer group coordinator detects timeout
  3. Partition reassigned to another consumer
  4. New consumer resumes from last committed offset (e.g. 4)
  5. Processes offset 4 and 5 again (duplicate)
  6. Commits offset 6 after success
Result: at-least-once by default. Exactly-once possible with transactional producers + idempotent consumers.

When to Use a Message Queue

A message queue is the right tool when your primary concern is distributing units of work to available workers and you don't need history, replay, or multiple independent consumers seeing the same data.

Transactional Email

User signs up. API puts "send welcome email" job in queue. Email worker picks it up, calls SendGrid, ACKs. You don't need to replay email jobs — once sent is once sent.

Celery + Redis / SQS + Lambda
Image / Video Processing

Upload triggers job: "resize to 3 sizes + generate thumbnail." Multiple workers share the queue. One job = one processing run. No need to re-process old uploads.

RabbitMQ + workers / SQS + EC2
Payment Processing

"Charge customer $99" job is sensitive — exactly one worker should process it. Queue semantics + idempotency key prevent double charges even on retry.

SQS FIFO + Lambda
REQUEST / REPLY PATTERN

RabbitMQ supports reply-to queues: the producer puts a temporary reply-queue address in the message. The consumer processes the job and publishes the result to that reply queue. The producer is unblocked waiting on its private reply queue. Useful for synchronous-feeling async operations (e.g. run a slow report, get back a presigned S3 URL).

WHEN YOU DON'T NEED A QUEUE

Don't over-engineer. If you have one producer and one consumer with low throughput, a queue may add unnecessary operational overhead. If you need replay, multiple subscribers, or audit history, skip the queue and use Kafka. If you need strict synchronous response, use HTTP/gRPC directly.

THE QUEUE DECISION CHECKLIST
  • Work needs to be distributed across N workers
  • Once processed, the job result is stored elsewhere
  • You don't need to replay old jobs
  • Only one consumer should handle each job
  • Failure handling via retry and DLQ is sufficient
  • Job-level priority is important (x-priority)

When to Use an Event Stream

An event stream is the right tool when the history of what happened is itself valuable, multiple independent consumers need the same data, or you need to rebuild state from scratch by replaying events.

EVENT SOURCING

Instead of storing current state in a database row, you store a sequence of events: AccountCreated, DepositMade($100), WithdrawalMade($30). Current balance = replay all events. Kafka is the persistence layer. Bugs in your state computation? Fix the code, replay, correct state falls out automatically.

MULTIPLE INDEPENDENT CONSUMERS

One order.placed event needs to trigger: inventory reduction, billing, shipping notification, analytics update, ML feature pipeline. Each is a separate consumer group reading the same stream independently. With a queue you'd need to manually fanout to 5 separate queues.

AUDIT LOGS AND COMPLIANCE

Kafka is append-only and durable. Write every sensitive action as an event: user.data.accessed, config.changed, payment.attempted. The log is tamper-evident (log compaction preserves history). Compliance teams query the stream. Impossible to "delete" an audit record — the immutability is the feature.

REPROCESSING AFTER BUG FIX

You discover a bug in your recommendation engine — it incorrectly classified 3 days of user behavior. With Kafka: fix the bug, reset your consumer group offset to 3 days ago, reprocess. The correct state is rebuilt automatically. With a queue: those messages are long gone — you have no history to replay.

Real-Time Analytics

Kafka Streams or Flink reads the event stream and computes running aggregates: revenue per minute, active users, error rates. State is the stream, not a DB table.

Uber, LinkedIn, Airbnb use this
Microservices Choreography

Services publish events, other services react. No orchestrator — no central brain. Order service publishes order.created. Inventory, billing, shipping all subscribe independently.

Loose coupling, high cohesion
CDC (Change Data Capture)

Debezium streams database changes as Kafka events. Every INSERT/UPDATE/DELETE becomes an event. Downstream systems stay in sync without polling the DB.

Postgres WAL + Debezium + Kafka

OUTBOX PATTERN

The classic distributed systems problem: you save to DB and publish to Kafka in the same "operation" — but they're not atomic. The DB might commit and Kafka publish fail (or vice versa). Solution: write to an outbox table in the same DB transaction. A CDC tool (Debezium) watches the outbox and streams changes to Kafka. DB and Kafka are now consistent.

SAGA PATTERN

Distributed transactions across microservices using a sequence of local transactions + compensating events. Orchestration saga: a central saga orchestrator sends commands to services via message queue and waits for replies. Choreography saga: each service publishes events to a stream; downstream services react. Kafka is better for choreography; RabbitMQ is better for orchestration.

CQRS + EVENT STREAM

Command Query Responsibility Segregation: writes go to the command side (events published to Kafka), reads go to a query side (a read-optimized materialized view built by consuming the stream). The event stream is the source of truth; read models are projections built from it. Invalidate and rebuild any read model by replaying the stream.
