The Spectrum of Consistency

Consistency in distributed systems is not a binary switch. It is a spectrum with many named points, each offering a different trade-off between safety guarantees and the cost of coordination. Stronger guarantees require more nodes to agree before returning a result — and that agreement takes time and can fail.

THE CORE TRADE-OFF

Every step stronger on the consistency spectrum requires more coordination. Linearizability demands that every read sees the most recent write globally — achieving that means querying a quorum or a leader on every read, which adds round-trip latency proportional to your network distance. In a global system spanning continents, that can mean 150–300ms of overhead per operation. Eventual consistency lets you read locally with no coordination — 1ms — but you might see stale data.

Eventual Consistency

The guarantee is simple: if no new writes arrive, all replicas will eventually converge to the same value. There is no guarantee on when. During the convergence window, different replicas may return different values for the same key. This is not a bug — it is an intentional design choice that maximises availability and performance.

Example: DNS Propagation

You add an A record for api.example.com pointing to your new server. The DNS root and your registrar update immediately. But:

  • Your local resolver: sees the new IP in 1 minute
  • Resolver in Singapore: sees it in 4 hours
  • Resolver in Brazil: sees it in 24–48 hours

During that window, half the world hits your old server, half hits the new one. Eventually — they all agree. Classic eventual consistency.

Example: Social Media Likes

A post has 1,000 likes. You and your friend open it simultaneously from different data centres:

  • You see: 1,000 likes (replica A, slightly lagged)
  • Friend sees: 1,001 likes (replica B, more up-to-date)

One second later, both replicas sync. You both see 1,001. The discrepancy was harmless — no one was harmed by seeing one fewer like momentarily.

AMAZON DYNAMO PAPER (2007)

Werner Vogels and the Amazon team published the Dynamo paper describing how they built their shopping cart using eventual consistency intentionally. The insight: "Is it better to show you an empty cart or a slightly stale cart?" A stale cart that still shows your items is far better UX than refusing to load the cart because a replica is unreachable. Availability wins over consistency for the shopping cart. This paper influenced the design of Cassandra, Riak, and DynamoDB.

Read-Your-Own-Writes (Session Consistency)

A targeted upgrade from eventual consistency: you always see your own writes, even if replication hasn't finished and other users might still see stale data. This is also called session consistency because the guarantee applies within a single client session.

THE FAILURE SCENARIO

User uploads a new profile photo. Write goes to Replica A. User refreshes the page — their next request is load-balanced to Replica B, which hasn't received the replication yet. User sees their old photo. They upload again, thinking it failed. Classic stale read after a self-write. Infuriating UX, even though the system is "correct" under pure eventual consistency.

Implementation Strategies

STICKY SESSIONS

Tag the client session with the replica they wrote to. Route all subsequent reads for that session to the same replica. Simple, but ties users to a node — creates hotspots.

READ FROM PRIMARY

After a write, read all requests from the primary for a 1-second window (the expected replication lag). After that window, allow reads from replicas again. Trade-off: primary becomes a bottleneck during that window.

CLIENT-SIDE TIMESTAMP

Client tracks the timestamp of its last write. Reads include that timestamp. Replicas refuse to answer if their data is older than the timestamp — forces routing to a sufficiently fresh node.

04

Causal Consistency

Causal consistency captures real-world cause-and-effect: if event B happened after event A, and B depends on A (B knows about A), then any node that sees B must also have seen A first. Concurrent events (neither knows about the other) can be seen in any order.

THE COMMENT REPLY PROBLEM

Alice posts a comment: "Who wants pizza tonight?". Bob replies: "Me! I love pizza!". Under eventual consistency without causal ordering, Bob's reply can arrive at a replica before Alice's original comment. The reply appears without the question — completely nonsensical to any reader hitting that replica. Causal consistency prevents this: any node that serves Bob's reply must have already applied Alice's comment, because Bob's write causally depends on Alice's.

Happens-Before Relationship

Event A happens-before event B (written A → B) if:

  • A and B are in the same process, and A came first in program order
  • A is a send, B is the corresponding receive of that message
  • There exists a chain: A → C → B for some event C

If neither A → B nor B → A holds, the events are concurrent and may be observed in either order.

Vector Clocks

Each node maintains a vector of counters, one per node in the cluster. On every event, it increments its own counter. On receiving a message, it takes the element-wise max with the sender's vector.

Node A writes: VC = [A:1, B:0, C:0]
Node B receives: VC = [A:1, B:1, C:0]
Node B replies: VC = [A:1, B:2, C:0]
B's reply causally follows A's write.
Any node serving B:2 must have A:1.
DYNAMODB CONDITIONAL WRITES

Amazon DynamoDB implements causal consistency through conditional expressions. A write can specify: condition_expression="version = :expected" — the write only succeeds if the current version in the database matches the client's expected version. This ensures a write that depends on seeing version N can only be applied after that version is visible, enforcing the causal dependency at the application level without vector clocks.

05

Linearizability (Strong Consistency)

The strongest and most intuitive guarantee: every operation appears to execute atomically at some single point in real time, and every read sees the most recent write from any client in the entire cluster. A linearizable system behaves identically to a single-server system with no concurrency — even if it is actually running on dozens of machines across multiple data centres.

What Linearizability Guarantees
  • If write W completes before read R starts, R must see W's value (or a later write)
  • If two writes are concurrent, all readers see the same order for both writes
  • There is a total order of all operations consistent with real-time order
  • No operation can go "back in time" — once a value is seen, no older value is ever returned
The Cost
  • Every read must contact a quorum of nodes (or the leader) to confirm the latest value
  • Network round-trip to multiple nodes on every read — cannot serve from local cache
  • If a quorum cannot be reached (network partition), the system must refuse the read
  • In a global system (US + EU + Asia), a linearizable read from Sydney might take 200ms
  • This is the CAP theorem's C: you must sacrifice availability (A) to maintain it under partition (P)

Real-World Linearizable Systems

System Linearizability Mechanism Use Case
ZooKeeper Leader-based consensus (ZAB protocol). All writes go through leader. Reads from leader are linearizable; reads from followers are sequentially consistent unless you call sync() first. Distributed coordination, leader election, config management.
Google Spanner TrueTime API: GPS clocks + atomic clocks give bounded clock uncertainty (±7ms). Commits wait out the uncertainty window before becoming visible. Globally consistent reads without a central leader. Google's globally distributed relational database. Financial systems.
etcd Raft consensus. All reads default to linearizable (routed through leader). Can opt for serializable reads for lower latency when linearizability is not required. Kubernetes cluster state, distributed locks, service discovery.
CockroachDB Serializable isolation + HLC (Hybrid Logical Clocks). Transactions are serializable, with external consistency (linearizability) achievable via AS OF SYSTEM TIME on reads. Globally distributed SQL transactions.

Quorum-Based Consistency (Cassandra / DynamoDB)

Quorum-based systems let you dial your own consistency level per operation by tuning three numbers: N (total replicas), W (write acknowledgements needed), and R (read responses needed). The magic formula: W + R > N guarantees that at least one node in your read set has the latest write.

Conflict Resolution in Eventual Consistent Systems

When two nodes accept writes for the same key during a network partition and then reconnect, you have a conflict: two diverged versions. How does the system decide what the "real" value is?

LAST WRITE WINS (LWW)

Compare timestamps. Whichever write has the higher timestamp survives; the other is discarded. Used by Cassandra as its default conflict resolution.

  • Pro: Simple, no application logic needed
  • Con: Data loss. If clocks are skewed, a newer write can lose to an older-timestamp write. Clock drift can be 100ms on real servers.
  • Con: Concurrent writes to different fields of the same row: all fields revert to the winning timestamp's values, even fields that weren't changed by either write
VECTOR CLOCKS (AMAZON DYNAMO)

Instead of timestamps, track the causal history of each value. When merging, if one version's history fully contains the other's, the superset wins (no true conflict). If the histories diverged (concurrent writes), both versions are kept and returned to the application to resolve.

  • Pro: No silent data loss — conflicts are surfaced
  • Con: Application must implement merge logic
  • Dynamo's shopping cart: merge by union — add all items from both versions
CRDTs (CONFLICT-FREE REPLICATED DATA TYPES)

Mathematical data structures where any two states can always be merged into a single consistent state — automatically, without conflicts. Used by Redis, Riak, collaborative editors.

G-Counter (grow-only):

APPLICATION-LEVEL (COUCHDB / RIAK)

The database detects a conflict and stores all versions. The application reads the conflicting revisions and decides how to merge them — e.g., a calendar app might union overlapping events, a bank account might sum the deltas.

  • Pro: Maximum flexibility and correctness
  • Con: Every application must implement domain-specific merge logic
