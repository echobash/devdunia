Why Eviction Exists: Bounded Memory vs Unbounded Data

A cache is fast memory — RAM, L1/L2/L3 CPU cache, Redis instance — and it is always bounded. Disk and network are "infinite" from the cache's perspective. The moment you try to store more data than fits in your cache, you have a problem: something must leave to make room for something new. The algorithm that decides which entry gets the boot is your eviction policy.

THE EVICTION DECISION

When the cache is full and a new entry must be inserted, the eviction policy answers: which existing entry has the lowest future value? The perfect answer (Belady's Algorithm) requires knowing the future — obviously impossible. Real policies approximate it using recency (LRU), frequency (LFU), insertion order (FIFO), or time limits (TTL). Each approximation trades correctness for practicality.

FIFO: First In, First Out

The simplest eviction policy: entries are evicted in the order they were inserted. The oldest entry — regardless of how frequently or recently it's been accessed — is always next to go. This is a queue: new entries join the back, evictions happen at the front.

FIFO STRENGTHS
  • Simplest possible implementation — a circular buffer or queue
  • O(1) insert and O(1) evict — no metadata overhead
  • Deterministic: eviction order is always predictable
  • Good for access patterns that are truly sequential with no re-access
FIFO WEAKNESS: BELADY'S ANOMALY

Belady's anomaly: with FIFO, adding more cache can cause MORE misses, not fewer. This seems impossible but is mathematically proven for FIFO. Example: page reference string 1,2,3,4,1,2,5 with 3 frames = 9 misses, with 4 frames = 10 misses. LRU and LFU don't have this problem — they are "stack algorithms" where more cache always means at least as many hits.

WHEN TO USE FIFO

FIFO works when entries truly have equal future value and you want the simplest possible expiry: job queues where older jobs should run first, log rotation where oldest segments roll off, event replay buffers. It is almost never the right choice for a data cache where access patterns show recency or frequency bias.

03

LRU: Least Recently Used

LRU evicts the entry that was accessed least recently — not the oldest insert, but the entry that has been sitting untouched the longest. The insight is temporal locality: if you used something recently, you're likely to use it again soon. What you haven't touched in a long time probably won't be needed.

DATA STRUCTURE: DOUBLY LINKED LIST + HASHMAP

The key engineering challenge is achieving O(1) for both get and set. A doubly linked list alone gives O(n) lookup. A hashmap alone can't maintain order. Together they solve both: the hashmap stores key → node pointer, the doubly linked list maintains recency order. On every access, move the node to the head in O(1). On eviction, remove the tail in O(1).

LRU STRENGTHS
  • Excellent for workloads with temporal locality (recently accessed = likely to be accessed again)
  • O(1) get and set with hashmap + doubly linked list
  • Stack algorithm: more cache always means fewer or equal misses
  • The de facto default for most production caches (Redis, Memcached, CPU TLB)
LRU WEAKNESS: SCAN RESISTANCE

A large sequential scan — reading 1 million cache keys once — will completely flush your working set. Every hot entry gets evicted because the scan entries briefly become the "most recently used". This is the cache pollution problem. Solutions: segmented LRU (S3-LRU), LIRS, or Redis's approximated LRU which samples randomly to avoid this.

04

LFU: Least Frequently Used

LFU tracks how many times each entry has been accessed and evicts the entry with the lowest total access count. The insight: if something has been requested thousands of times, it's probably worth keeping even if it hasn't been touched in a while.

DATA STRUCTURE: FREQUENCY MAP + MIN-HEAP

Store a freq_map[key] = count and a min-heap ordered by count. On get: increment count, re-heapify (O(log n)). On insert: if full, pop the minimum-count entry from the heap (O(log n)). A more efficient O(1) implementation uses a doubly linked list of frequency buckets — each bucket holds all keys with that frequency, and a pointer to the current minimum bucket.

LFU'S FREQUENCY BIAS PROBLEM

Imagine a video that goes viral for one hour and gets 10,000 cache hits. The next day, that video is irrelevant, but its frequency counter is 10,000. A new trending video with only 100 hits will be evicted before the stale video every single time. Solutions: aging (periodically halve all counters, used by Redis LFU with lfu-decay-time), or windowed LFU (count only accesses within a recent time window).

Redis Eviction Policies: All 8 Explained

When Redis reaches its maxmemory limit, it runs an eviction policy to free space. Redis offers 8 distinct policies covering different trade-offs between what gets evicted (all keys vs only keys with TTL) and how the victim is chosen.

maxmemory-policy <policy>  # set in redis.conf or via CONFIG SET
Policy What It Evicts How Victim Is Chosen Best Use Case
noeviction Nothing — returns error on write when full N/A When data loss is unacceptable. App must handle write errors explicitly.
allkeys-lru Any key in the keyspace Least recently used (approximate LRU via sampling) General-purpose cache. Most common default for Redis caches. Good locality properties.
volatile-lru Only keys that have a TTL set Least recently used among TTL keys Mix of persistent and cached data in one Redis instance. Protects non-expiring keys.
allkeys-lfu Any key in the keyspace Least frequently used (with frequency decay) Workloads with stable "hot" items accessed frequently. CDN-like patterns.
volatile-lfu Only keys that have a TTL set Least frequently used among TTL keys Frequency-based eviction without touching persistent keys.
allkeys-random Any key in the keyspace Random selection When all data has roughly equal access probability. Simplest possible eviction.
volatile-random Only keys that have a TTL set Random selection among TTL keys Simple expiry with protection for persistent keys. Low overhead.
volatile-ttl Only keys that have a TTL set Soonest-to-expire key evicted first When you want to expire imminently-expiring entries first, freeing longer-lived ones.
CHOOSING BETWEEN allkeys-lru AND volatile-lru

If Redis is a pure cache (all data is reproducible from a source of truth), use allkeys-lru — Redis can evict anything, maximizing hit rate. If Redis holds both persistent data (session state, queues) and cached data (computed results), use volatile-lru — only TTL-tagged cache entries are evictable, permanent data is safe.

REDIS LFU TUNING

Redis LFU uses a probabilistic counter (Morris counter) that saturates at 255, combined with a decay mechanism. Two config knobs: lfu-log-factor (how fast counter grows — higher = slower saturation, better discrimination at high frequencies) and lfu-decay-time (minutes until counter is halved — prevents frequency bias from old viral data). Default: factor=10, decay=1.

TTL-Based Expiration: Time Kills the Cache Entry

TTL (Time-To-Live) is not strictly an eviction policy — it's an expiration mechanism. Every entry has a countdown timer. When the timer hits zero, the entry is invalid. This ensures data freshness: stale entries can't live in the cache past a known deadline regardless of access patterns.

LAZY DELETION (CHECK ON ACCESS)

The simplest TTL approach: don't actively scan for expired entries. When a key is accessed, check if it's expired. If yes, delete it and return a miss. Pros: zero background CPU, no scan needed. Cons: expired entries sit in memory until someone asks for them. If a key expires but is never accessed again, it leaks memory indefinitely. Redis uses this as its first line of defense.

ACTIVE EXPIRATION (BACKGROUND SCAN)

Redis runs a background timer (default: every 100ms) that samples 20 random volatile keys, deletes expired ones, and if more than 25% of the sample expired, repeats immediately. This prevents memory bloat from keys that are never accessed again. The sampling is bounded to avoid blocking the event loop — Redis stays single-threaded and this scan is adaptive, not exhaustive.

HOW TTL INTERACTS WITH EVICTION: volatile-ttl

When Redis is full and uses volatile-ttl, it evicts keys with the shortest remaining TTL first — entries that are "about to expire anyway". This is efficient: instead of waiting for natural expiry, Redis proactively cleans up near-expired entries. The key insight: volatile policies only evict keys that have a TTL set; keys without TTL are never touched by eviction (though they will be by lazy/active expiration when their TTL fires).

Clock Algorithm: Approximate LRU Without the Overhead

True LRU requires updating a doubly linked list on every single cache access — a write to shared state that becomes a bottleneck in concurrent systems. The Clock Algorithm (also called Second Chance or NRU — Not Recently Used) approximates LRU using a single bit per entry and a circular buffer, with almost no write overhead on hits.

CLOCK DIAL — HAND SWEEPS CLOCKWISE
CLOCK
HAND
SECOND CHANCE ALGORITHM
ON CACHE ACCESS (HIT)
Set the entry's reference bit = 1.
That's it. No list manipulation, no write to shared structure.
ON EVICTION NEEDED
1. Clock hand points to current entry
2. If ref bit = 1: set bit = 0, advance hand (second chance given)
3. If ref bit = 0: EVICT this entry, insert new one here
4. Advance hand past inserted entry
WHY IT APPROXIMATES LRU
Entries used since the last clock sweep have bit=1 and get a second chance. Entries untouched since the last sweep have bit=0 and are the first evicted. This approximates "not recently used" without maintaining exact timestamps.
WHY REDIS USES APPROXIMATED LRU (5-KEY SAMPLING)

True LRU requires a global doubly linked list across all keys. In Redis (single-threaded event loop), updating this list on every GET would be prohibitively slow. Instead, Redis samples N random keys (default N=5, configurable via maxmemory-samples) and evicts the one with the oldest LRU clock. This is statistically good enough: sampling 10 keys is nearly indistinguishable from true LRU in practice, with 1/N the overhead. Higher maxmemory-samples = more accurate but more CPU per eviction cycle.

CLOCK IN OS PAGE REPLACEMENT

The Clock algorithm originated in operating system VM page replacement, where a hardware accessed bit (A-bit) is set automatically by the MMU on every memory access. The OS kernel sweeps a clock hand over physical page frames: if A=1, clear it and move on; if A=0, the page hasn't been touched since the last sweep — page it out. Linux uses a variant called the two-hand clock (active list + inactive list), which is what /proc/vmstat's pgactivate and pgdeactivate counters track.
