How S3 Is Architected Internally

S3 is not a simple file server. Under the hood it is a Dynamo-style distributed storage system — a consistent hashing ring of thousands of storage nodes, spread across multiple availability zones, with automatic replication and self-healing. Here is what happens when you call PutObject.

CONSISTENT HASHING RING
OBJECT PLACEMENT FORMULA
object_key = hash(bucket_name + "/" + key)
node_id = ring.lookup(object_key)
// Replicated on 3 nodes by default
// Adjacent nodes on the ring

The hash of bucket/key maps to a position on a virtual ring. The primary storage node for that position — and the two next nodes on the ring — each store a replica. This means no central directory is needed to find any object.

CONTENT-ADDRESSED BLOCKS

Large objects are split into content-addressed chunks. Each chunk's ID is the hash of its contents — identical chunks across different objects are stored only once. This provides free deduplication at the block level and makes integrity verification trivial (re-hash and compare).

ETag (MD5 of content) returned. Object durably stored.
S3 STANDARD: 3+ AVAILABILITY ZONES

Unlike a naive replication approach that stores 3 replicas in the same datacenter, S3 Standard distributes replicas across at least 3 separate Availability Zones within a region. A whole datacenter can burn down and your data survives intact. Each AZ has independent power, cooling, and networking — they are physically separated by miles.

The Object Data Model: Buckets, Keys, and Metadata

Understanding S3's data model precisely prevents a lot of confusion. There are no real folders. There is no rename. An "object" is not just a file — it is a bundle of data, metadata, and identity attributes.

THE FLAT NAMESPACE ILLUSION

When you see photos/2024/january/ in the S3 console, you are seeing a simulation. There is no directory object. There is no inode. The slash is just a character in the key string. When you "list a folder", S3 finds all keys with that prefix and groups them by the delimiter /. There is no rename operation — you copy the object to a new key and delete the old one.

METADATA: TWO KINDS

System metadata is set by S3: Content-Type, Content-Length, Last-Modified, ETag. You can influence some of these on upload.

User-defined metadata must start with x-amz-meta-. Maximum 2 KB total. Stored alongside the object and returned in GET response headers. Use it for tags, author info, processing state — anything your application needs without downloading the object itself.

Multipart Upload: Handling Giant Files Reliably

Uploading a 10 GB file in a single HTTP request is a bad idea. A network hiccup after 9.9 GB means starting over from scratch. Multipart upload solves this: split the file into independently uploadable parts, upload them in parallel, then tell S3 to assemble them. You only need to retry the one failed part.

PART SIZE RULES

Minimum part size is 5 MB (except the last part). Maximum part size is 5 GB. Maximum number of parts is 10,000. This means the maximum object size is 5 GB × 10,000 = 5 TB. The AWS SDK handles all of this automatically — it splits your file at the right boundaries and issues parallel UploadPart requests.

Presigned URLs: Direct Access Without Exposing Credentials

Your S3 bucket is private. But you need users to upload profile pictures or download invoice PDFs. You do not want to stream those gigabytes through your app server — that wastes your bandwidth and adds latency. Presigned URLs solve this by embedding temporary, scoped credentials directly in a URL.

S3 validates the signature on every request. If the expiry has passed or the URL has been tampered with, S3 returns 403.

GET vs PUT PRESIGNED URLS

GET presigned URL: lets a user download a specific object directly from S3 without authentication. Share it in an email, embed in an <img> tag — it works until expiry (1 min to 7 days). The object stays private — only holders of the URL can access it.

PUT presigned URL: lets a user upload to a specific key. Your server generates it, client PUTs directly to S3. Bandwidth cost: zero on your servers. You can constrain the allowed Content-Type and max object size with a presigned POST policy.

Versioning and Lifecycle Policies: Automate Your Storage Costs

S3 can automatically keep previous versions of every object, and it can automatically move objects to cheaper storage tiers as they age. Together these features mean you can set it and forget it — your old data costs a fraction of a cent per GB while still being recoverable.

Versioning

When versioning is enabled on a bucket, every PUT creates a new version rather than overwriting. Every DELETE creates a delete marker (a special version that hides the object without destroying it). Previous versions remain accessible by version ID.

Consistency Model and Durability: 11 Nines

S3 switched to strong consistency in December 2020. Before that, it was eventually consistent — you could write an object and immediately read a stale version back. Today, every successful PUT is immediately visible to subsequent reads. Let us also untangle the difference between durability and availability — they are often confused.

STRONG CONSISTENCY (DEC 2020)
  • After a successful PUT returns 200, any subsequent GET returns the new version — no stale reads
  • After a DELETE, any subsequent GET returns 404 — no ghost reads
  • List operations also strongly consistent — new objects show up immediately in listings
  • No extra cost or configuration — it is the default for all buckets
BEFORE 2020: EVENTUAL CONSISTENCY
// Overwrite PUT bug (pre-2020)
1. PUT object version A → 200 OK
2. PUT object version B → 200 OK
3. GET object → might return A!
// Replica not yet propagated

This forced developers to add retry logic, sleep-and-poll patterns, and version-check loops. All of that is unnecessary today.

Durability vs Availability

DURABILITY: 99.999999999%
= 1 object lost per 10 billion objects per year
HOW:
• 3+ replicas across 3+ AZs
• Erasure coding for large objects
• CRC checksums on every block
• Background scrubbing and healing
AVAILABILITY: 99.99%
= ~52 minutes downtime per year
MEANING:
• Your data is NOT lost during downtime
• S3 API may briefly return 503
• Durability and availability are separate SLAs
THE ERASURE CODING UPGRADE

S3 does not simply store 3 full copies anymore for large objects. It uses erasure coding (Reed-Solomon) — similar to RAID-6. Data is split into N chunks plus M parity chunks. Any N chunks are sufficient to reconstruct the object, even if M chunks are lost. This achieves higher durability with less storage overhead than naive 3x replication.
